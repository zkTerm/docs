## Overview

zkAuth provides passwordless authentication using social login (Google, GitHub, X) with STARK proofs on Starknet for verification. No passwords, no seed phrases - just seamless Web3 authentication.

**New in zkTerm Explorer:** All authenticated users can be explored in the zkTerm Explorer, a fully decentralized interface hosted on IPFS via Crust Network. Browse profiles, wallets, transactions, and activity - all powered by zkAuth.

<Note>
  zkAuth is powered by Web3Auth for key management and uses Cairo STARK proofs for quantum-resistant on-chain verification.
</Note>

## How It Works

<Steps>
  <Step title="Social Login">
    User authenticates with Google, GitHub, or X via Web3Auth.
  </Step>
  <Step title="Key Generation">
    Web3Auth generates a non-custodial key unique to the user and app.
  </Step>
  <Step title="STARK Proof">
    A STARK proof is generated proving ownership of the key without revealing it.
  </Step>
  <Step title="On-Chain Verification">
    The proof is verified on Starknet via a Cairo smart contract.
  </Step>
</Steps>

```bash

```

Clears session and revokes STARK proof.

```bash

```

Generates a new STARK proof for the current session.

## Technical Architecture

### Web3Auth Integration

zkAuth uses Web3Auth for non-custodial key management:

```typescript
import { Web3Auth } from "@web3auth/modal";

const web3auth = new Web3Auth({
  clientId: process.env.WEB3AUTH_CLIENT_ID,
  chainConfig: {
    chainNamespace: "other",
    chainId: "SN_SEPOLIA",
  },
});

await web3auth.initModal();
const provider = await web3auth.connect();
```

### STARK Proof Generation

STARK proofs are generated using Cairo:

```cairo
#[starknet::contract]
mod ZkAuthVerifier {
    use starknet::ContractAddress;
    
    #[storage]
    struct Storage {
        verified_users: LegacyMap<ContractAddress, bool>,
        proofs: LegacyMap<felt252, StarkProof>,
    }
    
    #[external(v0)]
    fn verify_auth(
        ref self: ContractState,
        public_key_hash: felt252,
        proof: Span<felt252>
    ) -> bool {
        // STARK verification logic
        let valid = verify_stark_proof(public_key_hash, proof);
        if valid {
            self.verified_users.write(get_caller_address(), true);
        }
        valid
    }
}
```

### Security Model

<Accordion title="Key Management">
  Web3Auth generates keys using threshold cryptography. The key is split between:
  - User's device (1 share)
  - Web3Auth network (1 share)
  - Recovery share (optional)
  
  No single party can access the full key.
</Accordion>

<Accordion title="Proof Expiry">
  STARK proofs have a configurable expiry (default: 7 days). After expiry, a new proof must be generated to access protected resources.
</Accordion>

<Accordion title="Session Binding">
  Each proof is bound to:
  - Specific session ID
  - User's public key
  - Timestamp
  - Application domain
</Accordion>

## Comparison with Traditional Auth

| Feature | Traditional | zkAuth |
|---------|-------------|--------|
| Password | Required | None |
| Seed Phrase | Required (Web3) | None |
| Privacy | Email exposed | ZK proof, no email on-chain |
| Security | Password attacks | STARK (quantum-resistant) |
| UX | Multiple steps | One-click |

### Privacy Considerations

<Warning>
  While zkAuth preserves privacy during authentication, the Explorer displays **public** on-chain data (wallet addresses, transaction hashes, zkID DIDs). Private data (passwords, files, proof inputs) remain encrypted and invisible.
</Warning>

**What's Public:**
- zkID DIDs (`did:zk:xxx`)
- Wallet addresses (Starknet, Solana)
- Transaction hashes and timestamps
- Compressed token counts
- IPFS CIDs (files are encrypted)

**What's Private:**
- Passwords and private keys
- File contents (AES-256-GCM encrypted)
- zkSNARK proof inputs
- Email addresses (unless user chooses to share)
